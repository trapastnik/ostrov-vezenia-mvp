# Остров Везения — Бизнес-процессы

> Документ для обсуждения с заказчиком. Описывает текущее состояние системы, выявленные вопросы и пробелы в логике.

---

## 1. Жизненный цикл заказа

### 1.1 Статусы заказа (12 шт.)

```
Принят → Ожидает забора → На складе → Формирование партии → На таможне →
→ Таможня пройдена → Ожидает отправки → Отправлен → В пути → Доставлен
```

Дополнительно: **Отменён** (тупик), **Проблема** (можно вернуть почти на любой этап).

### 1.2 Кто и что запускает переход

| Переход | Кто запускает | Как |
|---------|--------------|-----|
| → Принят | Система | Автоматически при создании заказа через API магазина |
| Принят → Ожидает забора | Оператор | Вручную в админке |
| Ожидает забора → На складе | Оператор | Вручную в админке |
| На складе → Формирование партии | Система | Автоматически при добавлении заказа в партию |
| Формирование партии → На таможне | Система | Каскад при смене статуса партии |
| На таможне → Таможня пройдена | Система | Каскад при смене статуса партии |
| Таможня пройдена → Ожидает отправки | Система | Каскад при смене статуса партии |
| Ожидает отправки → Отправлен | Оператор | Вручную в админке |
| Отправлен → В пути | Оператор | Вручную в админке |
| В пути → Доставлен | Оператор | Вручную в админке |
| Любой → Отменён | Оператор | Вручную (доступно до этапа «На таможне») |
| Любой → Проблема | Оператор | Вручную (доступно с этапа «На таможне») |

### 1.3 Создание заказа

При создании заказа через API магазина система:
1. Проверяет лимиты: **200 EUR** на посылку, **31 кг** на вес
2. Рассчитывает стоимость доставки через API Почты России
3. Добавляет таможенный сбор (настраивается для каждого магазина, по умолчанию 150 руб.)
4. Требует **паспортные данные** получателя (серия 4 цифры, номер 6 цифр)

## 2. Жизненный цикл партии

### 2.1 Статусы партии (4 шт., строго линейные)

```
Формирование → На таможне → Таможня пройдена → Отправлена
```

### 2.2 Процесс

1. Оператор открывает раздел «Партии» → «Создать партию»
2. Выбирает заказы в статусе «На складе» (чекбоксы)
3. Система создаёт партию, заказы переходят в «Формирование партии»
4. Оператор нажимает «На таможню» → все заказы в партии переходят в «На таможне»
5. «Таможня пройдена» → заказы в «Таможня пройдена»
6. «Отправить» → заказы в «Ожидает отправки»

### 2.3 Ограничения

- Нельзя отменить партию или вернуть на предыдущий статус
- Нельзя добавить/убрать заказы после создания партии

## 3. Жизненный цикл таможенной декларации (ДТЭГ)

### 3.1 Статусы (5 шт.)

```
Черновик ↔ Готова к подаче → Подана → Принята
                                    → Отклонена → Черновик
```

### 3.2 Процесс

1. Оператор выбирает заказы → создаёт декларацию
2. Заполняет таможенные поля товаров (код ТН ВЭД, страна происхождения)
3. Система валидирует (все поля заполнены, нет запрещённых товаров, лимиты)
4. Перевод в «Готова к подаче» → экспорт CSV/PDF
5. Оператор вручную подаёт документ в ФТС
6. Отмечает результат: «Принята» или «Отклонена»

### 3.3 Экспорт

- **CSV**: формат ДТЭГ для загрузки в таможенное ПО (разделитель `;`, кодировка UTF-8 BOM)
- **PDF**: печатная форма декларации (кириллица, формат A4)

## 4. Расчёт стоимости доставки

### 4.1 Формула

```
Итого = Тариф Почты России (публичный) + Таможенный сбор (настройка магазина)
```

### 4.2 Компоненты

| Компонент | Источник | Настраивается |
|-----------|----------|---------------|
| Тариф Почты | API Почты России (публичный тарификатор) | Нет (рассчитывается ПР) |
| Таможенный сбор | Настройка магазина `customs_fee_kopecks` | Да, для каждого магазина |

### 4.3 Лимиты

- Стоимость посылки: **до 200 EUR** (курс из ЦБ РФ)
- Вес: **до 31 кг**
- При превышении — доставка недоступна, заказ не создаётся

## 5. Группировка отправок (оптимизатор)

### 5.1 Идея

Объединение посылок по направлениям (хабам) для магистральной доставки. Экономия за счёт разницы между суммой публичных тарифов по отдельности и корпоративного тарифа на общий вес.

### 5.2 10 хабов

Москва, Санкт-Петербург, Екатеринбург, Новосибирск, Краснодар, Нижний Новгород, Казань, Ростов-на-Дону, Самара, Уфа.

### 5.3 Алгоритм (Celery, каждые 30 мин)

1. Берёт заказы в статусе `customs_cleared` без группы
2. Группирует по хабу (первые 3 цифры индекса)
3. Считает экономию = сумма публичных тарифов − корпоративный тариф
4. Отправляет если: есть просроченные ИЛИ (экономия > порога И заказов >= минимума)

### 5.4 Настройки

| Параметр | По умолчанию |
|----------|-------------|
| Макс. ожидание | 24 часа |
| Мин. заказов в группе | 3 |
| Мин. экономия | 500 руб. |
| Штраф за час задержки | 50 руб./час |

## 6. Интеграции

### 6.1 Плагин Bitrix (v0.3.0)

**Работает:**
- Служба доставки «Остров Везения» в checkout (расчёт через API)
- Паспортные поля в checkout (обязательные)
- Автоэкспорт заказа при оформлении (событие `OnSaleOrderSaved`)
- Сохранение ID заказа Остров в свойствах заказа Bitrix

**Не реализовано:**
- Трекинг статуса в ЛК покупателя
- Retry при падении API

### 6.2 Webhook-уведомления магазинам

**Инфраструктура готова** (Celery задача, подпись HMAC-SHA256), но **не подключена** — смена статуса заказа не отправляет уведомления.

### 6.3 API Почты России

**Используется:** расчёт тарифов (публичный + корпоративный).
**Не используется:** создание отправлений, получение трек-номеров, отслеживание.

## 7. ВОПРОСЫ ДЛЯ ОБСУЖДЕНИЯ

### 7.1 Связь партий и деклараций (КРИТИЧНО)

**Сейчас:** Партия и декларация — полностью независимые сущности. Заказ может быть в партии A и декларации B. Или в партии, но без декларации.

**Вопрос:** Должна ли декларация создаваться автоматически при формировании партии? Или декларация всегда = одна партия? Или допускается свободная комбинация?

**Варианты:**
- A) Партия = декларация (1:1, автоматическое создание)
- B) Декларация привязана к партии, но можно несколько деклараций на партию
- C) Свободная комбинация (как сейчас), но с предупреждениями

### 7.2 Два параллельных процесса после таможни (КРИТИЧНО)

**Сейчас есть два независимых механизма:**

1. **Партии**: После `customs_cleared` → партия `shipped` → заказы `awaiting_carrier`
2. **Оптимизатор группировки**: Ищет заказы в `customs_cleared` → создаёт группу → ставит `awaiting_pickup`

**Проблема:** Они конфликтуют. Партия переводит заказы в `awaiting_carrier`, а оптимизатор ищет `customs_cleared`. Если партия отработала первой — оптимизатор ничего не найдёт.

**Вопрос:** Как должен выглядеть процесс после прохождения таможни?
- A) Партия доводит до `awaiting_carrier`, далее оператор вручную отправляет
- B) После таможни оптимизатор группирует заказы по направлениям для магистральной доставки
- C) Партия — для таможни, оптимизатор — для доставки, они работают последовательно (партия → таможня → оптимизатор → отправка)

### 7.3 Влияние статуса декларации на заказы

**Сейчас:** Смена статуса декларации (принята/отклонена) **никак не влияет** на статусы заказов.

**Вопрос:** Нужно ли?
- При отклонении декларации → заказы в «Проблема»?
- При принятии → что-то менять?

### 7.4 Webhook: какие события отправлять магазину?

**Варианты:**
- A) Все смены статуса
- B) Только ключевые: отправлен, доставлен, проблема
- C) Настраивается для каждого магазина

### 7.5 Создание отправлений в Почте России

**Сейчас:** Система только рассчитывает тариф. Само отправление в Почту России **не создаётся** через API. Трек-номер ПР не присваивается.

**Вопрос:** Нужно ли автоматическое создание отправлений через API ПР? Или это делается вручную в системе Почты?

### 7.6 Статусы «В пути» и «Доставлен»

**Сейчас:** Оператор вручную меняет статус на «В пути» и «Доставлен». Нет интеграции с трекингом Почты России.

**Вопрос:**
- Нужен ли автоматический трекинг через API ПР?
- Или достаточно статуса «Отправлен» как конечного с точки зрения Остров Везения?

### 7.7 Отмена и возвраты

**Сейчас:** «Отменён» и «Доставлен» — тупиковые статусы, из них нельзя выйти.

**Вопрос:** Нужен ли процесс возврата? Может ли доставленный заказ быть возвращён?

### 7.8 Редактирование заказа после создания

**Сейчас:** После создания заказа нельзя изменить адрес, получателя, состав. Можно только редактировать таможенные поля товаров (код ТН ВЭД, страна).

**Вопрос:** Нужна ли возможность редактирования? До какого статуса?

### 7.9 Модель ценообразования

**Сейчас:** Фиксированный таможенный сбор за посылку (по умолчанию 150 руб., настраивается для каждого магазина).

**Вопрос:** Это финальная модель? Или нужно:
- За товар (а не за посылку)?
- Процент от стоимости?
- Прогрессивная шкала?
- Разные ставки для разных категорий?

### 7.10 Изменение состава партии

**Сейчас:** После создания партии нельзя добавить или убрать заказы.

**Вопрос:** Нужна ли возможность редактирования партии в статусе «Формирование»?

## 8. НАЙДЕННЫЕ ТЕХНИЧЕСКИЕ ПРОБЛЕМЫ

> Эти проблемы не требуют бизнес-решений, будут исправлены в рамках разработки.

### 8.1 Два разных курса EUR

Расчёт доставки использует курс из `.env` (статичный), а декларации — курс из базы данных (обновляется из ЦБ). Нужно унифицировать.

### 8.2 Оптимизатор обходит валидацию переходов

Оптимизатор меняет статус заказа напрямую (`order.status = "awaiting_pickup"`), минуя функцию `change_order_status()`. Из-за этого не создаётся запись в истории статусов.

### 8.3 Номер партии — риск коллизии

Номер формата `B-YYYYMMDD-HHMMSS` (секундная точность). При одновременном создании двух партий — конфликт. Нужен суффикс.
